// What is MatrixReview

Your docs become
your reviewer.

Every engineering team writes documentation. Security policies, architecture standards, style guides, contribution rules. MatrixReview reads all of it, then enforces it on every PR. Automatically.

Generic AI code review tools give opinions based on general best practices. MatrixReview gives document-backed findings. Every flag references your company's own documentation, not someone else's standards.

On setup, MatrixReview clones your repo, builds a full import dependency graph (nodes, edges, entry points), and discovers all your documentation in one pass. When a PR arrives, it computes the blast radius of your changes, identifies every upstream and downstream file affected, and pulls only the relevant docs for review. No context flooding. Focused analysis.

Findings are tagged DOC-BACKED or AI SUGGESTION so your team knows exactly what's policy and what's observation. Doc-backed findings cite the specific document name and line range. If it can't be proven from your docs, it doesn't get posted.

// Review Gates

Five gates. Zero guesswork.

Every PR is reviewed across five specialized gates. Each gate pulls only the documents relevant to its category and reviews the diff against that subset.

GATE:SECURITY

Security

API keys, auth patterns, secrets in code, data exposure, injection risks. Reviewed against your security policies and incident response docs.

GATE:ARCHITECTURE

Architecture

Design patterns, module boundaries, dependency rules, API contracts. Catches architectural drift before it becomes tech debt.

GATE:LEGAL

Legal & Compliance

Licensing, CLA requirements, copyright headers, redistribution compliance. Every commit checked against your legal requirements.

GATE:STYLE

Style & Standards

Naming conventions, formatting, linting, import order, code standards. Your style guide enforced automatically. Not someone else's opinion.

GATE:ONBOARDING

Onboarding & Process

PR process, testing requirements, commit format, contribution workflow. New contributors stay in compliance from their first PR.

SYSTEM:VERIFICATION

Three-Pass Verification Pipeline

Every finding goes through a three-pass pipeline before it reaches the PR. Pass 1 generates findings by reviewing the diff against your documentation with blast radius context. Pass 2 is a separate verification model that re-reads each finding against the source document and kills anything unproven. Pass 3 is a dedicated citation sweep that searches your docs for the exact filename and line range supporting each finding.

If a finding can't survive all three passes, it never reaches your PR comment. The system monitors its own accuracy and flags gates with high removal rates for prompt tuning.

SYSTEM:FRESHNESS

Auto-Updating Knowledge Base

Docs stay fresh automatically. Someone updates a security policy, merges a new architecture decision, adds a testing requirement. MatrixReview detects the change via SHA comparison before the next review and re-ingests the updated content.

No stale rules. No re-setup. The knowledge base tracks the repo. If a document can't be fetched during the freshness check, the review runs with cached docs. Stale docs are better than no review.

// Deep Dive

How it actually works.

Under the hood, MatrixReview is a fail-closed execution system. Here's what happens from the moment you install to the moment findings land on your PR.

01 INSTALLATION &
DISCOVERY

Install once. Graph builds itself.

When you install MatrixReview on a repository, the system clones your repo, walks the entire filesystem, and builds a full import dependency graph. Nodes are your files, edges are your imports, and entry points are identified automatically. On cal.com (25,000+ files), this completes in under 60 seconds.

During the same walk, MatrixReview discovers all your documentation files and classifies them into review gates using parallel LLM analysis. 300+ docs classified in about 13 seconds. Multi-topic docs (like a CONTRIBUTING.md with security rules and style guides in one file) are decomposed into gate-specific sections with exact line ranges.

No configuration files. No YAML. No rule authoring. Install the GitHub App, open a PR, and setup runs automatically on the first trigger. Under 2 minutes for a 25,000 file repo.

02 DOCUMENT
DECOMPOSITION

One doc, multiple gates. Handled surgically.

Real-world documentation doesn't follow neat categories. A CONTRIBUTING.md might contain security policies, style rules, and PR process requirements all in one file. The decomposer handles this.

Step 1: LLM identifies distinct topical sections with line ranges. Step 2: Content is extracted using those line ranges. Deterministic extraction, not LLM regeneration. The AI identifies where sections are; code pulls the content by line number. This means the extracted text is exactly what was written, never a paraphrase or hallucination.

03 THE REVIEW
PIPELINE

Five gates, three passes, zero hallucinations shipped.

When a PR arrives, MatrixReview loads the dependency graph and computes the blast radius: every file changed plus every upstream and downstream import affected. It then pulls only the documentation relevant to those files and their gates. Not 300 docs. 10 to 20 focused, relevant docs.

Pass 1 runs five parallel gate reviews, each with lane-disciplined prompts (Security only sees security issues, Architecture only sees architecture issues). Pass 2 is a separate verification model that re-reads each finding against the source docs and kills anything that contradicts or can't be proven. Pass 3 is a citation sweep that searches the docs for exact line references for every surviving finding.

Cross-gate dedup consolidates duplicates by priority (Security > Architecture > Legal > Style > Onboarding). The output is a traffic light: RED, YELLOW, or GREEN. Every finding shows the source document, line range, and whether it's doc-backed or an AI suggestion.

04 FAIL-CLOSED
EVERYWHERE

Every decision path has a fallback.

MatrixReview is built on a fail-closed architecture. If the system can't prove a finding from your documentation, the finding doesn't ship. If GitHub is unreachable, the review degrades gracefully and posts an error comment rather than failing silently. If a document can't be fetched during freshness check, the review runs with cached docs. Stale docs are better than no review.

Structured logging with JSON output and request IDs runs through the entire pipeline. Every scan, classification, decomposition, review, and verification step is traceable. When something goes wrong, you can follow the exact path of execution. When something goes right, you can prove it.
