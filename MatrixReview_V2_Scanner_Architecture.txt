================================================================================
SECTION 1: SCANNER CELL ARCHITECTURE
================================================================================

The scanner follows the SGI cell pattern:

  CellScanner:
    INIT:    Load language patterns, CWE detectors, import resolver
    PROCESS: Scan every file, produce fingerprints
    CERTIFY: Validate fingerprints (no empty fields, no unresolved refs)
    REPORT:  Return scan manifest + per-file fingerprints

The scanner is a single cell with internal sub-modules:

  1. FileWalker        — Traverses repo tree, filters, classifies file types
  2. StructureExtractor — Extracts imports, exports, definitions per language
  3. ImportResolver     — Resolves import statements to actual file paths
  4. RiskClassifier     — Detects vulnerability patterns (CWE-backed)
  5. DependencyMapper   — Builds the file dependency graph
  6. GateRelevanceScorer — Maps files to doc review gates
  7. ManifestBuilder    — Assembles final output artifacts

SECTION 3: RISK CLASSIFIER — RESEARCH-BACKED DETECTORS
================================================================================

Like NCE's 9 mechanisms drawn from neuroscience, the risk classifier has
detectors drawn from CWE vulnerability research. Each detector is:
  - Deterministic (regex or AST pattern, no LLM)
  - CWE-mapped (linked to the specific weakness it detects)
  - Gate-assigned (which review gate it feeds into)
  - Point-valued (contributes to the scoring system)
  - Evidence-producing (captures the exact line and pattern)

Starting with Python. Each detector is independently testable and validated.

────────────────────────────────────────────────────────────────────────────────
SECURITY DETECTORS (Gate: SECURITY)
────────────────────────────────────────────────────────────────────────────────

DETECTOR S1: HARDCODED_SECRETS
  CWE: CWE-798 (Use of Hard-coded Credentials)
  Points: 15 (CRITICAL)
  Research: Most prevalent Python weakness per CWE studies. Easy to detect
  statically with high confidence. Hard-coded credentials in source are the
  #1 cause of credential leaks.
  Patterns:
    - Variable names matching: password, secret, api_key, token, private_key,
      access_key, auth_token, bearer, credentials, passphrase
    - Assigned to string literals (not env vars, not function calls)
    - Exclude: test files, example files, documentation
  Confidence: 0.95 (very high because the pattern is unambiguous)
  False positive mitigation: Ignore assignments from os.environ, config
  objects, function return values. Only flag literal string assignments.

DETECTOR S2: SQL_INJECTION
  CWE: CWE-89 (Improper Neutralization of Special Elements in SQL)
  Points: 12 (HIGH)
  Research: Top 3 most exploited vulnerability type. F-string and format()
  in SQL queries are the primary Python vector.
  Patterns:
    - f-string containing SQL keywords: SELECT, INSERT, UPDATE, DELETE, DROP
    - .format() on strings containing SQL keywords
    - String concatenation (+) with SQL keywords and variables
    - % formatting with SQL keywords
  Confidence: 0.90
  False positive mitigation: Ignore parameterized queries (?, %s placeholders
  in the query string with separate params argument).

DETECTOR S3: COMMAND_INJECTION
  CWE: CWE-78 (Improper Neutralization of Special Elements in OS Command)
  Points: 12 (HIGH)
  Research: Python's subprocess and os.system are primary vectors.
  Patterns:
    - os.system() with variable interpolation
    - subprocess.call/run/Popen with shell=True and variable args
    - eval() with external input
    - exec() with external input
  Confidence: 0.85
  False positive mitigation: subprocess with shell=False and list args is safe.

DETECTOR S4: PATH_TRAVERSAL
  CWE: CWE-22 (Improper Limitation of a Pathname)
  Points: 10 (HIGH)
  Research: Common in web applications that serve files.
  Patterns:
    - open() with user-supplied path without sanitization
    - os.path.join() with user input without checking for ".."
    - send_file / send_from_directory with user-supplied filename
  Confidence: 0.75 (higher false positive rate, needs context)

DETECTOR S5: INSECURE_CRYPTO
  CWE: CWE-327 (Use of Broken Crypto Algorithm)
  Points: 8 (MEDIUM)
  Research: MD5 and SHA1 for security purposes are broken.
  Patterns:
    - hashlib.md5() (except when explicitly for checksums, not auth)
    - hashlib.sha1() for password or token hashing
    - DES, RC4, ECB mode usage
    - random.random() used for security tokens (not cryptographically secure)
  Confidence: 0.85

DETECTOR S6: INSECURE_DESERIALIZATION
  CWE: CWE-502 (Deserialization of Untrusted Data)
  Points: 12 (HIGH)
  Research: pickle.loads on untrusted data allows arbitrary code execution.
  Patterns:
    - pickle.loads() / pickle.load() on network/user input
    - yaml.load() without Loader=SafeLoader
    - marshal.loads()
  Confidence: 0.90

DETECTOR S7: HARDCODED_URLS
  CWE: CWE-798 (variant)
  Points: 5 (MEDIUM)
  Research: Hardcoded HTTP URLs (not HTTPS) indicate insecure communication.
  Internal staging URLs leaked in production code.
  Patterns:
    - http:// URLs in source code (not test, not localhost)
    - Staging/dev URLs in non-test files
  Confidence: 0.70

DETECTOR S8: MISSING_ERROR_HANDLING
  CWE: CWE-755 (Improper Handling of Exceptional Conditions)
  Points: 6 (MEDIUM)
  Research: Bare except clauses and broad exception catching mask errors.
  Patterns:
    - except: (bare except)
    - except Exception: (overly broad) without logging
    - No try/except around network calls, file I/O, database operations
  Confidence: 0.65 (context-dependent, higher false positive rate)

DETECTOR S9: SENSITIVE_DATA_EXPOSURE
  CWE: CWE-200 (Exposure of Sensitive Information)
  Points: 8 (MEDIUM)
  Research: Logging or printing sensitive data.
  Patterns:
    - print() or logging.* containing variables named password, token, secret
    - Exception messages that include full stack traces with user data
    - Debug endpoints that expose internal state
  Confidence: 0.70

DETECTOR S10: UNSAFE_REGEX
  CWE: CWE-1333 (Inefficient Regular Expression Complexity)
  Points: 4 (LOW)
  Research: ReDoS attacks via catastrophic backtracking.
  Patterns:
    - Nested quantifiers: (a+)+, (a*)*
    - Alternation with overlap: (a|a)+
    - re.compile() on user-supplied patterns without timeout
  Confidence: 0.80

SECTION 4: POINT SYSTEM AND SCORING
================================================================================

POINT VALUES BY SEVERITY:
  CRITICAL: 15 points
  HIGH:     10-12 points
  MEDIUM:   5-8 points
  LOW:      2-4 points
  INFO:     1 point

SCORE FORMULA:
  raw_points = sum of all risk signal points across all files
  normalized_points = raw_points / sqrt(code_line_count)
    (sqrt normalization so large codebases aren't penalized linearly)
  score = max(0, 10 - (normalized_points / scale_factor))
    where scale_factor is calibrated so that:
      0-2 points normalized = 10/10 (pristine)
      3-5 points normalized = 8-9/10 (healthy)
      6-10 points normalized = 6-7/10 (needs attention)
      11-20 points normalized = 4-5/10 (significant issues)
      21-40 points normalized = 2-3/10 (critical state)
      40+ points normalized = 0-1/10 (emergency)

PER-GATE SCORES:
  Each gate gets its own score using the same formula but only
  counting points from that gate's detectors.

OVERALL SCORE:
  Weighted average with worst-case drag:
    overall = weighted_avg(gate_scores) - penalty(worst_gate)
    where penalty = max(0, (5 - worst_gate_score) * 0.5)
  This means a 2/10 security score drags the overall down
  even if all other gates are 9/10. Same principle as MR-1
  worst-case aggregation.

PR SCORE EVOLUTION:
  When a PR is reviewed:
    new_findings_points = sum of new finding points
    fixed_findings_points = sum of previously flagged issues now resolved
    delta = new_findings_points - fixed_findings_points
    new_score = recalculate(previous_raw_points + delta)

  Each PR stores its delta. The dashboard shows:
    "PR #142 by [author]: +14 points (security +12, architecture +2)"
    "PR #145 by [author]: -10 points (fixed security issue)"

SECTION 6: IMPORT RESOLVER (PYTHON)
================================================================================

Full import resolution for Python. Deterministic. No LLM.

RESOLUTION ORDER (mirrors Python's actual import machinery):
  1. Check if import is stdlib (use a static list of stdlib modules)
  2. Check if import is a known third-party package (from requirements.txt)
  3. Resolve relative imports using file position in package hierarchy
  4. Resolve absolute imports by:
     a. Search from project root
     b. Follow __init__.py chains
     c. Handle "from X import Y" by resolving X to a file, then Y to
        a name within that file
  5. If all else fails, mark as "unresolved" with the raw import string

OUTPUTS PER IMPORT:
  - raw: the original import statement
  - resolved_path: the file it points to (or null)
  - resolved: boolean
  - stdlib: boolean
  - external: boolean (third-party package)
  - names: list of specific names imported

STDLIB LIST:
  Maintain a static set of Python 3.10+ stdlib module names.
  ~300 modules. Updated per Python version.

THIRD-PARTY DETECTION:
  Parse requirements.txt, setup.py, setup.cfg, pyproject.toml for
  declared dependencies. Any import matching a declared dependency
  is marked external.

SECTION 7: BUILD PLAN
================================================================================

Each sub-module is built, tested, and certified independently.
Same embryonic development approach as SGI cells.

STEP 1: FileWalker
  - Input: repo root path
  - Output: list of file paths with metadata (size, extension, role)
  - Test: run against C:\CodeReview, verify correct file discovery
  - Certify: every file in the repo is accounted for (included or skipped)

STEP 2: StructureExtractor (Python)
  - Input: single Python file contents
  - Output: imports, definitions, exports
  - Test: run against known Python files, verify extraction accuracy
  - Certify: compare against manual inspection of 10 files

STEP 3: ImportResolver (Python)
  - Input: list of imports + project file tree
  - Output: resolved paths for each import
  - Test: run against C:\CodeReview, verify resolution accuracy
  - Certify: spot-check 20 imports manually

STEP 4: RiskClassifier
  - Input: single file contents + structural extraction
  - Output: risk signals with CWE mappings and points
  - Test: create test files with known vulnerabilities, verify detection
  - Certify: zero false negatives on test suite, <10% false positive rate

STEP 5: DependencyMapper
  - Input: all resolved imports across all files
  - Output: adjacency list, cycle detection, fan-in/fan-out per file
  - Test: run against C:\CodeReview, verify graph accuracy
  - Certify: manual verification of critical path identification

STEP 6: GateRelevanceScorer
  - Input: file path, structural extraction, risk signals
  - Output: per-gate relevance scores (0-1)
  - Test: verify auth files score high on SECURITY, test files on ONBOARDING
  - Certify: compare against manual classification of 20 files

STEP 7: ManifestBuilder
  - Input: all fingerprints + dependency graph + scores
  - Output: scan_manifest.json
  - Test: full end-to-end scan of C:\CodeReview
  - Certify: manifest is valid, complete, and matches manual spot checks

STEP 8: Integration
  - Wire all modules together
  - Run full scan against 3 test repos (C:\CodeReview, cal.com, stripe)
  - Performance target: <5 seconds for 500 files, <30 seconds for 5000 files

SECTION 8: FUTURE EXPANSION
================================================================================

LANGUAGE EXPANSION:
  Each new language requires:
    1. Language-specific regex patterns for StructureExtractor
    2. Language-specific import resolution rules
    3. Language-specific risk patterns (some CWEs are language-specific)
  Everything else (FileWalker, DependencyMapper, ManifestBuilder,
  scoring) is language-agnostic.

  Priority order: Python (first), JavaScript/TypeScript, Go, Java, Rust

EMBEDDING LAYER:
  After the scanner is certified, the fingerprints feed into an
  embedding layer that vectorizes the structural map into hyperspace.
  Files become vectors. Proximity encodes structural relationships.
  Claude can query the hyperspace for related code without reading
  every file. This is the bridge between the deterministic layer
  and the LLM intelligence layer.

COALITION INTEGRATION:
  The scanner output feeds into the coalition aggregator. When
  CodeRabbit or Copilot flags a file, the scanner's risk profile
  for that file provides context: "CodeRabbit flagged auth_service.py.
  Our scanner already identified CWE-798 and CWE-89 in that file.
  CodeRabbit's finding aligns with our deterministic detection.
  Trust score for this finding: HIGH."
